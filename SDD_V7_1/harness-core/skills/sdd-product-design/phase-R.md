# 阶段 R：竞品调研

## 前置条件

- 当前项目为 Web 项目
- `docs/PRD.md` 尚不存在
- 用户已提供基本的产品方向描述

## 开始前必须先做

先检查是否已有场景对齐稿：

```text
docs/澄清文档/<feature-name>/01-alignment.md
```

如果没有，且用户输入仍是大段想法、口述需求或复杂业务描述，先调用：

```text
harness-core/skills/alignment/SKILL.md
```

产出并确认业务 PRD 对齐稿后，再进入竞品调研。

如果用户已经清晰确认产品目标、范围、核心流程、权限边界和外部依赖，可以直接进入本阶段。

## 澄清问题

提问必须覆盖：

- **用户与场景**：谁在用、什么场景、当前怎么解决的
- **核心痛点**：现有方案最大的问题
- **产品形态**：Web/移动/桌面、AI/传统
- **产品边界**：功能范围、角色权限、数据可见性、生成 / 保存 / 下载规则

不要询问数据库字段、接口设计、组件拆分、开发任务顺序等技术实现问题。

## 调研执行

充分理解后，调研同类竞品（至少 2 个，至少 1 个国内产品），输出：

- 竞品对比：关键流程、核心交互、视觉风格、优缺点、可借鉴点
- 参考方案 A/B/C（每套包含：页面结构、关键交互、视觉方向）

## 产出方式

调研结论和参考方案**在对话中呈现**，不单独持久化为文件。用户确认选用方案后，关键结论写入后续的 PRD.md 背景章节。

如果前置场景对齐稿存在，阶段 R 的调研结论必须继承其业务目标、范围和验收口径，不得自行扩大功能范围。

## 门禁

向用户发起确认：「请确认采用 A/B/C 还是融合方案；确认后进入阶段 A。」

**未确认不得进入阶段 A。**
