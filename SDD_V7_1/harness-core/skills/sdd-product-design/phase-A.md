# 阶段 A：页面 → 功能 → 版本规划 → PRD

## 固定产出

`docs/PRD.md`（持续迭代，本阶段结束时为初稿状态，阶段 C 补充定稿）。

PRD.md 包含：页面清单、主功能定义与分析、版本规划（Mission / Persona / V1·MVP / V2+）、业务规则、复杂链路、数据契约清单。

**执行顺序（禁止跳步）**：先确定「有哪些页面」，再基于页面梳理「每个页面做什么功能」，最后才落到「分几版做、哪些数据规则」。

```
A1 页面梳理 → A2 主要功能定义与分析 → A3 版本规划
→ A4 关键功能实现思路 → A5 数据契约确认
→ 落盘 PRD.md → 总确认
```

## A1：页面梳理（必须先做）

基于阶段 R 已选方向，先与用户对齐**产品全貌的页面结构**：

1. **列出所有页面**（或关键视图），包括：
  - 页面名称（用户可理解的自然语言命名）
  - 页面类型（列表页 / 详情页 / 表单页 / 仪表盘 / 对话页 等）
  - 可见角色（谁能看到这个页面）
2. **页面跳转逻辑**：
  - 从哪来、到哪去
  - 主要入口（导航菜单 / 按钮跳转 / 链接）
3. **全局布局结构**（如涉及）：
  - 是否有统一侧边栏 / 顶部导航
  - 公共模块（用户信息、通知、搜索）分布在哪

**输出格式约束**：

```markdown
### 页面清单

| 序号 | 页面名称 | 页面类型 | 可见角色 | 入口来源 | 跳转去向 |
|------|---------|---------|---------|---------|---------|
| P01 | 登录页 | 表单页 | 未登录用户 | 直接访问 | 首页/仪表盘 |
| P02 | 仪表盘 | 仪表盘 | 管理员 | 侧边栏 | 详情页 |

### 页面跳转图

```text
登录页 → 仪表盘 → 详情页
  ↓
注册页
```

```

**门禁**：用户明确确认「页面清单和跳转逻辑」后，才能进入 A2。

## A2：主要功能定义与分析（在 A1 确认之后）

**基于 A1 的每个页面**，逐页定义其承载的功能。

### 输出格式约束

```markdown
### 页面：P02 仪表盘

#### 功能清单

| 功能编号 | 功能名称 | 一句话描述 | 用户可感知的完成标准 |
|---------|---------|-----------|-------------------|
| F-02-01 | 数据概览 | 展示核心指标卡片 | 用户进入页面即看到3-5个实时数据卡片 |
| F-02-02 | 趋势图表 | 展示近7天数据趋势 | 用户可选择时间范围，图表随之更新 |

#### 功能边界（做什么 / 不做什么）

| 功能编号 | 包含 | 不包含 |
|---------|------|--------|
| F-02-01 | 展示总用户数、订单数、收入 | 不展示明细列表、不支持导出 |
| F-02-02 | 折线图展示趋势 | 不支持对比多个维度、不支持下钻 |

#### 与其他页面的关联

- 点击数据卡片 → 跳转 P03 明细列表页（携带筛选参数）
- 点击趋势图数据点 → 弹出当日详情弹窗（不跳转新页面）
```

**执行规则**：

1. 按页面顺序逐个分析，不要跨页面混排
2. 每个功能必须有明确的「用户可感知的完成标准」（用户怎么知道这个功能有了）
3. 功能编号规则：`F-{页面序号}-{功能序号}`，如 `F-02-01`

**门禁**：用户确认「所有页面的功能清单和边界」无歧义后，才能进入 A3。

## A3：版本规划 — Roadmap（在 A2 确认之后）

在**页面结构和功能定义**清楚的前提下，划分：

- **V1 / MVP**：哪些页面、哪些功能、做到什么深度（与 A2 定义一致）
- **V2+**：哪些页面或功能延后、为什么（依赖、成本、风险、非核心）
- 写入 **Mission、Persona、关键业务逻辑 (Business Rules)**，与上述切分自洽

**输出格式约束**：

```markdown
### 版本规划

#### V1 / MVP

| 页面 | 包含功能 | 排除功能 | 理由 |
|------|---------|---------|------|
| P01 登录页 | F-01-01 账号密码登录 | F-01-02 第三方登录 | 第三方登录需要申请Key，阻塞MVP |
| P02 仪表盘 | F-02-01 数据概览 | F-02-02 趋势图表 | 图表库引入增加复杂度，V2再做 |

#### V2+

| 功能 | 预计版本 | 延后理由 |
|------|---------|---------|
| F-01-02 第三方登录 | V1.1 | 需要微信/钉钉开放平台审核 |
| F-02-02 趋势图表 | V1.2 | 需要引入图表库，当前MVP以表格展示替代 |
```

**规则**：版本划分必须能回答「为什么这一块在 MVP、那一块在 V2」——依据应是实现复杂度、依赖链、风险，而不是随意罗列。

**门禁**：用户确认版本规划与对功能的取舍无误后，才能进入 A4。

## A5：数据契约确认

PRD 初稿中所有**影响流程结果**的数据定义，必须以 checklist 形式列出，由用户逐项确认。
**注意，整个项目的数据契约部分，不跟随V1或是V2变化，即使只是开发部分功能，数据契约也要完整设计**
### A5-1：业务数据契约

```markdown
### 数据契约确认清单

#### 商品分类体系
- [ ] 一级分类：服装、数码、食品、家居（共 4 类）
- [ ] 二级分类：服装-男装、服装-女装、数码-手机、数码-电脑...（共 15 类）

#### 用户画像字段
- [ ] 昵称（string）、注册时间（datetime）、会员等级（枚举：普通/白银/黄金）

#### 订单状态流转
- [ ] 状态：待支付 → 已支付 → 已发货 → 已签收 → 已完成

#### 权限角色
- [ ] 管理员：全部权限
- [ ] 普通用户：浏览、下单、查看订单
```

### A5-2：接口响应格式契约（必须在此阶段锁死）

```markdown
#### 统一响应格式
- [ ] 成功响应：`{"code": 200, "message": "success", "data": { ... }}`
- [ ] 错误响应：`{"code": <错误码>, "message": "<错误描述>", "data": null}`
- [ ] 分页响应：`{"code": 200, "data": {"items": [...], "total": 100, "page": 1, "page_size": 20}}`

#### HTTP 状态码约定
- [ ] 200：成功  /  400：参数错误  /  401：未认证
- [ ] 403：无权限  /  404：资源不存在  /  500：服务器内部错误
```

**规则**：业务数据契约和接口响应格式契约**全部与用户确定**后才能视为阶段 A 文档定稿。
