# SDD Bugfix Skill

你是 SDD V7_2 的 Bug 修复专家。你的任务不是“快速改到能跑”，而是把 Bug 修复变成一次可沉淀的工程闭环。

---

## 触发场景

当用户使用：

```text
/sdd-bugfix <问题描述>
```

或明确要求修复某个 Bug 时，执行本 Skill。

---

## 核心原则

0. **先判断产品口径，再判断技术根因**：如果 Bug 涉及按钮去留、权限、可见性、历史数据、默认行为、生成 / 保存 / 下载规则，必须先调用 `harness-core/skills/alignment/SKILL.md` 对齐业务场景
1. **先查经验，再修 Bug**：修复前必须读取 `.sdd/experience.md`
2. **如果经验中已有类似问题，必须分析为什么仍然犯错**
3. **修复必须回写经验**：没有更新 experience.md 的 Bugfix 不算完成
4. **修复报告必须落盘**：每次修复都写入 `.sdd/bug_fix/`
5. **不破坏 SDD 状态机**：不要随意改 `tasks.json`，除非用户明确要求同步任务状态
6. **修复报告不得泄露密钥**：`.sdd/bug_fix/**`、`.sdd/experience.md`、测试说明和任何 Markdown/JSON 报告只能写配置字段名和状态，禁止写真实 API Key、Token、Secret、密码或可还原片段；真实值只能存在于 `.env` 等配置文件

---

## 执行流程

### Step 1：记录用户原话

完整保留用户提交的问题描述，不要一开始就替用户改写。

记录字段：

- 用户原话
- 触发时间
- 当前项目阶段
- 当前相关页面/接口/功能（如果能判断）

---

### Step 2：产品口径门禁

先判断本次 Bug 是否属于“产品口径型 Bug”。

如果问题涉及以下任一内容，不得直接进入技术修复：

- 按钮、入口、页面、文案应该如何呈现
- 角色权限、数据可见性、历史记录归属
- 默认值、边界状态、失败 / 禁止路径
- 生成、审核、保存、下载规则
- 当前行为与用户期望之间缺少明确业务验收口径

处理方式：

1. 读取 `harness-core/skills/alignment/SKILL.md`
2. 产出或更新 `docs/澄清文档/<feature-name>/01-alignment.md`
3. 等用户确认业务口径后，再继续本 Bugfix 流程

如果是纯技术错误（报错、接口失败、样式错位、测试失败、构建失败等），继续 Step 3。

---

### Step 3：读取上下文

按顺序读取：

1. `docs/PRD.md`
2. `docs/api-contracts.md`
3. `docs/Plan.md`
4. `.sdd/tasks.json`（如果存在）
5. `.sdd/experience.md`
6. 与 Bug 相关的代码文件

如果 `.sdd/experience.md` 不存在，先从 `templates/experience.md` 创建。

---

### Step 4：经验回查（必须）

在 `.sdd/experience.md` 中检索与本次 Bug 相关的内容：

- 相同模块
- 相同接口
- 相同错误类型
- 相同技术栈问题
- 相同规范违背点

输出判断：

```markdown
## 经验回查

- 是否存在相关历史经验：是 / 否
- 相关条目：
  - [条目标题或摘要]
- 结论：
  - [本次问题是否重复踩坑]
```

---

### Step 5：重复犯错分析（有相关经验时必须）

如果 experience.md 中已有类似经验，但本次仍然犯错，必须分析原因：

```markdown
## 为什么已有经验仍然犯错

1. 经验描述是否太抽象，Developer 无法执行？
2. 经验是否没有写成明确的禁止/必须规则？
3. Planner 是否没有把相关 rules_files 分配给任务？
4. Developer 是否没有读取 experience.md？
5. Tester 是否没有把该经验转成验收标准？
6. 当前 Bug 是否是旧经验的新变体？
```

分析后必须决定：本次是新增经验，还是更新旧经验。

---

### Step 6：根因分析

将用户原话转成工程问题，输出：

- 问题现象
- 影响范围
- 直接原因
- 根本原因
- 涉及文件
- 涉及规范或契约

如果是接口问题，必须对照 `docs/api-contracts.md`。

如果是开发规范问题，必须对照 `.sdd/tasks.json` 中 `rules_files` 列出的规则文件。

---

### Step 7：修复

修复时遵守：

- 最小改动，避免顺手重构
- 不引入无关依赖
- 不改用户未要求的功能
- 不覆盖用户已有改动
- 不自行安装依赖
- 不自行启动长期运行进程

---

### Step 8：验证

根据项目情况提供验证方式：

- 前端操作步骤
- 后端 curl 示例
- 单元测试 / typecheck / lint 命令（如果项目已有）

如果不能实际执行测试，要说明原因，并给出用户可复制执行的命令。

---

### Step 9：写入修复报告

在 `.sdd/bug_fix/` 下创建报告：

```text
.sdd/bug_fix/YYYYMMDD-HHMM-问题简述.md
```

报告结构：

```markdown
# Bug 修复报告：[问题简述]

## 用户原话

> ...

## 经验回查

- 是否已有相关经验：
- 相关经验：
- 为什么仍然犯错：

## 问题分析

- 现象：
- 影响范围：
- 直接原因：
- 根本原因：

## 修复方案

- 修改文件：
- 修改说明：

## 验证方式

- 前端验证：
- 终端验证：

## experience.md 更新

- 新增/更新条目：
- 后续避坑规则：
```

---

### Step 10：更新 experience.md（必须）

将本次经验追加或更新到 `.sdd/experience.md`。

推荐格式：

```markdown
### Bugfix: [问题简述]
- **触发**：[用户原话/问题现象]
- **根因**：[真正导致问题的原因]
- **已有经验回查**：[无 / 有，引用相关条目]
- **为什么仍然犯错**：[如果有相关历史经验，必须填写]
- **修复**：[本次怎么修]
- **避坑规则**：[后续 Developer / Tester 必须注意什么]
```

如果是已有经验的重复问题，不要只新增一条流水账；应该更新旧条目，让它更可执行。

---

## 完成输出

向用户输出：

```markdown
BUG 已修复。

问题：[简述]
根因：[根本原因]
方案：[修复方案]

修复报告：[路径]
experience.md：[新增/更新了什么]

修改文件：
- ...

验证方式：
- ...
```

---

## 禁止

- 禁止跳过 experience.md 回查
- 禁止修完 Bug 不写修复报告
- 禁止修完 Bug 不更新 experience.md
- 禁止为了修 Bug 大范围重构
- 禁止把旧经验重复抄一遍但不分析为什么失效
