## 核心变更

### 1. 权限模型简化：3 角色，薪资权岗位自带

**旧模型**：3 角色 + 独立的 `payroll_access` 权限位（可单独授予）

**新模型**：3 角色，薪资权随角色自带，不单独授予

| 角色 | 薪资访问权 | 系统配置 | 业务数据读写 | 复盘/工单 |
|------|-----------|---------|------------|-----------|
| **技术超管** | ❌ 永久不可 | ✅ | ❌ | ✅ 运维视角 + 我的工单 |
| **业务超管** | ✅ **岗位自带**（每次查需二次确认+留痕） | ❌ | ✅ | ✅ 复盘 + 只读追踪 |
| **普通员工** | ❌ 永久隔离 | ❌ | ✅ | ❌ |

### 2. 薪资访问控制

#### 技术超管 / 普通员工
- 薪资分类在左侧导览**隐藏**
- 薪资字段**脱敏**（显示 `***`）
- 智能体问个人薪资在 **Planner 意图识别阶段直接 reject**

#### 业务超管
- 薪资分类/字段**可见**
- 展开薪资字段 **或** 智能体问个人薪资时：
  1. **先弹二次确认**（填事由）
  2. **写审计日志**（谁/何时/对象/字段/入口/事由）
  3. 再显示/作答

#### 技术超管审计页
- 可见"谁查了什么"的**访问元数据**
- **永远看不到任何薪资金额**

### 3. 复盘隐私边界

复盘报告**只消费全量聚合元信息**：
- 意图分布、badcase 原因比例、👍👎 聚合率、趋势
- **绝不展示个人逐条问答原文**
- 👍👎 是聚合统计，不是点名
- **复盘是看系统健康度，不是监控员工**

### 4. 工单流转

- **负责人固定 = 技术超管**，不可指派他人
- **业务超管对自己提的工单**：
  - 状态=**待处理** → 可**编辑/撤回**
  - 状态=**处理中及以后** → 只能**加备注**
- 工单"已完成"后**自动触发 ③Test 门禁**
- 门禁**绿 + 技术超管二次确认**才"已上线"
- **任何人不能跳过门禁**

## 主要代码变更

### 后端

#### 1. `backend/src/services/rbac.py`

```python
def has_effective_payroll_access(role: str, payroll_access: bool = False) -> bool:
    """新规格：薪资权=业务超管角色自带。
    
    技术超管、普通员工、agent 永久无薪资权。
    payroll_access 参数保留用于向后兼容。
    """
    role = normalize_role(role)
    if role == BIZ_SUPER_ADMIN:
        return True
    return False

def can_view_payroll_category(role: str, payroll_access: bool = False) -> bool:
    """只有业务超管可看薪资分类导览。"""
    return normalize_role(role) == BIZ_SUPER_ADMIN

def should_reject_personal_salary_query(role: str, payroll_access: bool = False) -> bool:
    """普通员工/技术超管问薪资直接 reject；业务超管需二次确认。"""
    return normalize_role(role) != BIZ_SUPER_ADMIN
```

## 注意事项

1. **复盘 Agent 后端本阶段不实现**：
   - 复盘报告页和工单页用 **mock 数据**
   - mock 数据放在清晰可替换位置（`MOCK_REVIEW_REPORT`）
   - 预留接口位，将来接真实复盘 Agent 只换数据源
   - **但权限、显隐、工单状态流转、审计留痕按真实逻辑实现**

2. **工单编辑/撤回/备注**：
   - 前端实现了按钮和 mock 交互
   - 后端接口预留（`PATCH /admin/tickets/{id}`，`POST /admin/tickets/{id}/notes`）
   - 需后续实现完整的工单编辑流程

3. **移交机制**：
   - 创建业务超管后，`must_change_password = True`
   - 首次登录强制改密（前端逻辑已有，需确保生效）
