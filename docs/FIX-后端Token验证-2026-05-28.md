# 后端薪资Token验证修复 - 2026-05-28

## 问题根因

Claude 老师分析确认：**前端正确带上 token + JWT 已标明薪资权限，但后端仍要求确认导致死循环。**

### 证据

用户提供的请求头显示：
- `x-payroll-confirm: HL12iIABv4iwAtRB9jzaPLRM6CGeWssI` ✅ token 已带上
- JWT payload: `role: biz_super_admin, payroll_access: true` ✅ 权限正确

但智能体持续返回 "需要二次确认"，导致 ask 请求无限循环（已 61 次）。

### 异常信号

**Token 每次都在变**：几次请求的 token 不同（IqrLLF→qLR9L8→HL12iI），说明后端每次都在生成新 token，而不是校验前端带回来的 token。

### 6. 异常情况

如果仍然循环，根据日志判断：

- **token 为空**：前端未正确传递 `X-Payroll-Confirm` header（前端问题）
- **token 不存在于内存**：
  - 后端重启导致内存清空
  - 前端传的 token 不是后端生成的（前端/后端不同步）
  - Token 过期（超过 30 分钟）
- **username 不匹配**：JWT 的 username 与 token 绑定的 username 不一致（不应该发生）

## 已知限制

### Token 内存存储的问题

当前 token 存储在 **Python 进程内存**（`_confirm_tokens` 字典），存在以下限制：

1. **不跨实例**：如果有多个 API 实例（负载均衡），token 不共享
2. **不持久化**：API 重启后所有 token 丢失，用户需重新确认
3. **不跨容器**：Docker 容器重启或重新部署会清空

### 长期解决方案（未实现）

可选的持久化方案：
1. **Redis**: 将 `_confirm_tokens` 存到 Redis，支持分布式、TTL 自动过期
2. **数据库**: 存到 `payroll_confirm_tokens` 表（需新建），手动清理过期
3. **JWT**: 将确认状态编码到 JWT claim（需刷新 token）

当前权衡：
- **内存方案** 简单，适合单实例开发环境
- **生产环境** 如需高可用（多实例/重启不影响），建议接入 Redis
