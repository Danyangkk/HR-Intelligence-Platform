## 🔍 根因分析

通过代码审查和流程追踪，发现了**三个关键问题**：

### 问题1：智能体薪资确认参数传递错误

**位置**: `frontend/index.html` - `handlePayrollConfirmForAgent` 函数

**错误代码**:
```javascript
const ok = await ensurePayrollAccess({
  target_ref: '智能体查询',
  entry: '个人薪资明细'  // ❌ 错误：entry 应该是 '智能体'
});
```

**正确代码**:
```javascript
const ok = await ensurePayrollAccess({
  target_ref: '个人薪资查询',  // 智能体查询的对象
  entry: '智能体',  // ✅ 入口
  fields: '薪资'  // ✅ 字段
});
```

**影响**:
- 后端接收到错误的 `entry` 值（`'个人薪资明细'` 而不是 `'智能体'`）
- 审计表记录的入口字段错误
- 可能导致后端校验或处理逻辑异常

### 问题2：API请求未携带薪资确认Token

**位置**: `frontend/index.html` - `apiHeaders` 函数

**问题描述**:
智能体重新发送请求时，虽然 `AUTH.payrollConfirmToken` 已经被设置，但 `apiHeaders()` 函数没有自动将其添加到请求头中。后端无法验证用户已经完成了薪资确认，导致请求被拒绝。

**错误代码**:
```javascript
function apiHeaders(extra={}, opts={}){
  const h = {...extra};
  if(!opts.skipJson) h['Content-Type'] = 'application/json';
  if(AUTH.token) h['Authorization'] = 'Bearer '+AUTH.token;
  // ❌ 缺少：if(AUTH.payrollConfirmToken) h['X-Payroll-Confirm'] = AUTH.payrollConfirmToken;
  return h;
}
```

**正确代码**:
```javascript
function apiHeaders(extra={}, opts={}){
  const h = {...extra};
  if(!opts.skipJson) h['Content-Type'] = 'application/json';
  if(AUTH.token) h['Authorization'] = 'Bearer '+AUTH.token;
  if(AUTH.payrollConfirmToken) h['X-Payroll-Confirm'] = AUTH.payrollConfirmToken;  // ✅ 添加
  return h;
}
```

**影响**:
- 智能体请求后端时，`X-Payroll-Confirm` 头缺失
- 后端无法验证用户已确认薪资访问
- 请求被拒绝，返回"需要二次确认"错误
- 前端再次触发弹窗，形成死循环

### 问题3：弹窗取消按钮未正确处理Promise

**位置**: `frontend/permission-admin.js` + `frontend/index.html`

**问题描述**:
用户点击弹窗"取消"按钮时，调用的是 `closeModal('payrollConfirmModal')`，只是关闭了弹窗UI，但没有调用 `payrollConfirmResolver(false)` 来通知 Promise 已取消。导致 `ensurePayrollAccess` 函数一直等待，程序卡住。

**错误代码**:
```html
<!-- 弹窗HTML -->
<button class="btn" onclick="closeModal('payrollConfirmModal')">取消</button>
<!-- ❌ 问题：closeModal 只关闭UI，不处理 Promise -->
```

**正确代码**:
```javascript
// permission-admin.js 中添加专门的取消函数
function cancelPayrollConfirm(){
  global.closeModal('payrollConfirmModal');
  if(payrollConfirmResolver) payrollConfirmResolver(false);  // ✅ 通知 Promise 已取消
  payrollConfirmResolver = null;
}
global.cancelPayrollConfirm = cancelPayrollConfirm;
```

```html
<!-- 弹窗HTML -->
<button class="btn" onclick="cancelPayrollConfirm()">取消</button>
<!-- ✅ 正确处理取消逻辑 -->
```

**影响**:
- 用户点击"取消"后，Promise 永不 resolve
- `handlePayrollConfirmForAgent` 卡住，不返回任何结果
- 用户无法继续使用智能体

## 🎯 技术要点

### 1. Promise 处理规范
```javascript
// ✅ 正确：确保 Promise 在所有情况下都能 resolve 或 reject
function openModal(){
  return new Promise(resolve => {
    resolver = resolve;
    // 确认按钮：resolver(true)
    // 取消按钮：resolver(false)
    // 关闭×：resolver(false)
  });
}

// ❌ 错误：只处理确认，不处理取消
function openModal(){
  return new Promise(resolve => {
    resolver = resolve;
    // 确认按钮：resolver(true)
    // 取消按钮：没有调用 resolver ← 永不返回！
  });
}
```

### 2. 自定义请求头自动注入
```javascript
// ✅ 好的做法：在统一的 apiHeaders 函数中处理
function apiHeaders(extra={}, opts={}){
  const h = {...extra};
  if(AUTH.token) h['Authorization'] = 'Bearer ' + AUTH.token;
  if(AUTH.payrollConfirmToken) h['X-Payroll-Confirm'] = AUTH.payrollConfirmToken;
  return h;
}

// ❌ 不好：每次请求都手动添加
fetch(url, {
  headers: {
    'Authorization': 'Bearer ' + AUTH.token,
    'X-Payroll-Confirm': AUTH.payrollConfirmToken,  // 容易遗漏
  }
});
```

### 3. 参数传递规范
```javascript
// ✅ 正确：参数语义明确，符合后端API定义
ensurePayrollAccess({
  target_ref: '个人薪资查询',  // 访问对象
  entry: '智能体',            // 入口
  fields: '薪资'              // 字段
});

// ❌ 错误：参数语义混乱
ensurePayrollAccess({
  target_ref: '智能体查询',
  entry: '个人薪资明细'  // entry 应该是入口，不是对象
});
```

## 📝 总结

这是一个**关键的流程性BUG**，涉及前端异步Promise处理、HTTP请求头传递、以及后端API参数规范。主要问题在于：

1. **参数传递错误**：智能体入口的 `entry` 字段传错，导致后端处理异常
2. **请求头缺失**：`X-Payroll-Confirm` 头没有自动添加，导致后端无法验证用户已确认
3. **Promise未正确关闭**：弹窗取消时没有调用 resolver，导致程序卡住

通过这三处修复，确保了智能体薪资查询的完整流程正常运行，用户现在可以：
- ✅ 在智能体中查询个人薪资
- ✅ 填写事由并确认后，正常获得答案
- ✅ 点击取消时，流程正常结束
- ✅ 审计表正确记录所有访问（入口="智能体"）
