# 前端页面规格 · 权限重构与改进闭环（给 Cursor）

> 本文是权限重构 + 改进闭环涉及的前端页面规格，供 Cursor 在现有 `index.html` 基础上修改，后端同步实现权限校验。配套《系统提示词与组件规范》《路由总纲 ROUTER》《AgentHarness/TestHarness/反馈采集 规格》《复盘Agent设想存档》。
>
> 总原则：**生产级实现**（作品集但贴合真实合规）。核心是**职责分离**——管系统的人（技术超管）永久碰不到业务/薪资数据；能看薪资的人（业务超管）不碰系统。**前端隐藏 ≠ 后端放行，所有权限后端必须二次校验。**

---

## 1. 权限模型（3 角色，无独立薪资权限位）

| 角色 | 看薪资 | 系统配置/建账号 | 业务数据读写 | 复盘/工单 |
|---|---|---|---|---|
| **技术超管**（开发/运维，即"我"） | ❌ **永久不可** | ✅ | ❌ | ✅ 运维视角 + 我的工单工作台 |
| **业务超管**（HRD） | ✅ **岗位自带**（每次查二次确认+留痕） | ❌ | ✅ | ✅ 看复盘 + 只读追踪工单 |
| **普通员工** | ❌ **永久隔离** | ❌ | ✅（可增删改导入） | ❌ |

**关键规则：**
- **薪资访问权随"业务超管"角色自带**，不需要也无法单独授予——取消了 payroll_access 权限位、取消发证页。能看个人薪资的全系统只有业务超管。
- **技术超管永久无薪资权**：UI 与后端都不给技术超管任何薪资数据访问能力（审计页能看"谁查了薪资"的元数据，但永远看不到薪资金额本身）。
- **普通员工永久薪资隔离**：左侧导览薪资分类不显示、薪资字段脱敏、智能体问个人薪资在**意图识别阶段直接拒答**。
- **薪资合规靠"二次确认 + 全程留痕审计"**，不靠授权动作（业务超管岗位自带权限，但每次访问留痕）。
- **工单负责人固定 = 技术超管**，不可指派他人。
- **复盘/工单仅技术超管与业务超管可见**，普通员工不参与运营复盘。
- **创建业务超管需移交**：技术超管建号后该账号首次登录强制改密，移交后技术超管无法登录该账号。

### 3.5 复盘报告页（按角色分两个视图；普通员工不可见）

> **核心原则：业务超管=决策者（能点采纳/驳回/存疑、有存疑待办、有提醒）；技术超管=执行者/观察者（纯只读，无任何决策按钮、无存疑待办、无提醒，但能看完整技术详情据以干活）。** 决策权归业务超管，技术超管只看健康度+据被采纳的工单干活。

#### 3.5.A 业务超管版（决策者）

> **业务超管版只展示"业务摘要层"，通篇业务语言，不出现技术术语。** 业务超管是决策者，要据此判断"严不严重、值不值得修"，看不懂技术黑话就无法决策。具体：
> - 问题归因 → 展示 **biz_problem（业务语言）+ impact（影响面/频次）+ priority（优先级）**。
>   - ❌ 错误（技术语言）："15例RAG 0命中集中在年终奖核算口径"
>   - ✅ 正确（业务语言）："员工问'年终奖怎么算'，系统答不上来 · 本周15次 · 优先级高"
> - 改进建议 → 展示**业务语言的目标**，不展示技术实现。
>   - ❌ "Planner few-shot 增加「人均」类聚合示例 / PLANNER_FEW_SHOT"
>   - ✅ "让系统能正确回答'人均成本'类问题"
> - **隐藏**所有技术字段（phenomenon/root_cause_hypothesis/node_clues/run_id/具体文件名/RAG/over_reject/Planner 等术语）——这些只在 3.5.B 技术超管版显示。
> - **改进建议同样分两版**：业务超管看 `content_biz`（业务语言，如"让系统能正确回答'各部门成本'类问题"）+ 采纳/驳回/存疑按钮；**不显示** draft_changes 的技术细节（ROUTER/§3/文件名/测试用例）。技术超管(3.5.B)看 draft_changes（改哪个文件、加什么测试）。
> - 前端直接展示复盘 Agent 输出的 biz_problem/impact/priority 字段（复盘 Agent 提示词已要求 biz_problem 用业务语言、不含技术术语，见复盘规格§5）。

**每条建议的三个操作 + 处理后状态标签：**
- 未处理 → 显示三个按钮：`[采纳→生成工单] [驳回] [存疑]`。
- 已处理 → 按钮区变为**状态标签**（建议不消失，永远留在报告里）：
  - 采纳后 → `✓ 已生成工单 #012 →`（可点跳工单详情）
  - 驳回后 → `✕ 已驳回`（可查看驳回理由）
  - 存疑后 → `⏸ 已存疑`
- 这样业务超管一眼能看出每条建议"处理过没、什么结果"，不重复纠结。

**右侧"存疑待办"区块（仅业务超管可见，常驻）：**
- 汇总**所有报告**里被存疑的建议（跨周），带角标显示数量。
- 每条显示：建议摘要 + 来自哪周 + 已挂起多久 + `[重新采纳] [驳回]` 两个操作。
- 重新采纳 → 生成工单（hold→accepted）；驳回 → 已驳回（hold→rejected，填理由）。
- 一直挂着，不过期。技术超管不可见此区块（存疑是业务超管的待决策，技术超管只看工单）。

**复盘提醒（复盘 Agent 生成时自动标注）：**
- 本周复盘生成时，复盘 Agent 语义比对**历史存疑** finding，若本周某 finding 与某条存疑高度相似 → 在该 finding 上标注：`⚠ 此问题上周已存疑，本周又发生N次，建议重新评估`。
- 只比对存疑的（不比对驳回的）。判定靠语义相似，不靠关键词。

**隐私边界（重要）**：复盘是**全量聚合的模式与统计**，**绝不展示某个用户问了什么的逐条原文**，👍/👎 也是聚合统计。复盘是看系统健康度，**不是监控员工**。问题原文在 trace 中以 hash 存储，复盘 Agent 只消费聚合元信息。

采纳弹层：
```
采纳建议 → 生成改进工单
 建议: ROUTER给aggregate补"成本类部门查询"边界
 依据: 8个badcase run [展开]
 工单草稿:
  - 改动点: 路由总纲ROUTER §3 aggregate判定
  - 须加测试: tests/router_cases.yaml 新增1条
  - 负责人: 技术主管(固定,不可改)
 ⚠ 仅生成待办工单，不修改任何配置。改动须人工完成并过门禁后上线。
                                    [取消] [生成工单]
```
- **驳回**：弹窗填驳回理由 → 建议显示"✕ 已驳回"标签留档（也是对复盘 Agent 归因质量的反馈）。**不做汇总、不做提醒**（驳回的不进存疑待办、不触发复盘提醒）。
- **存疑**：建议显示"⏸ 已存疑"标签 + 进入右侧存疑待办区块，可重新采纳/驳回，一直挂着。

| 元素 | 业务超管版(决策者) | 技术超管版(只读) |
|---|---|---|
| 量化概览 | ✅ | ✅ |

**工单操作边界（业务超管对自己提交的工单）：**
- **去掉"编辑"操作**（业务超管不改工单主体——工单内容来自复盘 Agent，改了会失真）。业务超管要补充信息一律用"加备注"。
- **"加备注"在所有状态可用**：待处理/处理中/待验证/已上线都能加备注。备注是**追加式留言**（每条带提交人+时间），不覆盖工单主体内容；技术超管在工单详情能看到所有备注。
- **"撤回"仅"待处理"可用**（技术超管未接单才能撤销该工单）。
- 已驳回 → 仅"详情"。
- 操作汇总：待处理 = 详情/加备注/撤回；处理中及以后 = 详情/加备注；已驳回 = 详情。
- 所有备注/撤回均留痕（谁、何时、内容）。业务超管对该页其他人提的工单 = 只读。

**从复盘页跳转 + 返回（双向打通）：**
- 业务超管在复盘页点"✓已生成工单#X→" → 跳到本页并**自动定位/打开 #X 的工单详情**；技术超管点同样链接 → 跳到"我的工单"工作台并定位该工单（按角色跳各自视图）。
- 工单详情页顶部有 **"← 返回复盘报告"**，点击回到来源复盘页（尽量回到原滚动位置）。通用原则：凡从 A 跳转/下钻到 B 的页面，B 都提供"← 返回 A"。

## 4. 工单状态流转（守门禁红线）

```
[复盘建议] ──采纳──> 待处理 ──[技术主管接单]──> 处理中
                       │(业务超管可编辑/撤回)        │(锁定主体,业务超管只能加备注)
处理中 ──[技术主管:已完成改动+加用例]──> 待验证
待验证 ──[系统自动跑③测试门禁]──>
        ├─ 绿 + 技术主管二次确认 ──> 已上线 ✅ (回填关联 badcase=fixed)
        └─ 红 ──> 退回处理中(附失败用例)
[复盘建议] ──驳回──> 已驳回(记理由) ; ──存疑──> 存疑(挂起,可重新激活)
```
**铁律：任何人（含技术主管本人）都不能跳过测试门禁直接上线**。"已完成"→自动门禁→绿+确认→才"已上线"。

## 5. 实现要点（给 Cursor）

1. **角色精简为 3 个**：技术超管/业务超管/普通员工。**取消 payroll_access 权限位与发证页**——薪资访问权随业务超管角色自带。
2. **薪资拦截/脱敏改为按角色**：
   - 技术超管、普通员工：薪资分类在左侧导览隐藏、字段脱敏、智能体问个人薪资拦截。其中**普通员工问个人薪资在 Planner 意图识别阶段直接 reject**（沿用现有薪资拦截逻辑，不弹任何确认）。
   - 业务超管：薪资分类/字段可见；**展开薪资字段 或 智能体问个人薪资时，先弹二次确认(填事由)+写审计，再显示/作答**。
   - 不删现有薪资拦截代码，只为业务超管加"二次确认+留痕后放行"的分支。
3. **后端按角色二次校验**：前端隐藏不代表后端放行；普通员工/技术超管直接调薪资接口须被后端拦。
4. **技术超管永久无薪资权**：硬编码，UI 与后端都不给技术超管薪资数据访问；审计页可见但不含金额。
5. **移交机制**：技术超管创建业务超管后，该账号首次登录强制改密；移交后技术超管无法登录该账号。
6. **审计留痕**：所有薪资访问（数据中台 + 智能体两个入口）写独立审计日志（谁/何时/对象/字段/入口/事由）。
7. **复盘隐私边界**：复盘只消费**全量聚合元信息**（意图分布、各 badcase 原因比例、👍/👎聚合率、趋势），**绝不展示个人逐条问答原文**；问题在 trace 中以 hash 存储。复盘是看系统健康度，不是监控员工。
8. **工单**：负责人固定技术主管；业务超管对自己提的工单在"待处理"可编辑/撤回，"处理中"后只能加备注；工单"已完成"后**自动触发 ③Test 门禁**，绿+二次确认才"已上线"。
9. **复盘 Agent 后端本阶段不实现**（定时任务/读trace/LLM归因是后续阶段）。复盘报告页与工单页**先用 mock 数据**把界面与交互（看报告、采纳生成工单、状态流转、过门禁）做出来，但 mock 数据放在清晰可替换的位置、预留数据接口位，将来接真实复盘 Agent 只换数据源、不重写页面。**权限、显隐、工单状态流转、审计留痕这些必须按真实逻辑实现，不能是 mock。**
