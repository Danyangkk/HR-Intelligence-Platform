# Fix: 智能体薪资确认死循环修复

**日期**: 2026-05-28 18:20  
**类型**: P0 严重Bug修复  
**状态**: ✅ 已修复并部署

---

## 🐛 问题描述

### 现象
业务超管在智能体中查询个人薪资时，填完确认表格后：
- ❌ 控制台不停刷屏："开始处理薪资确认 → 确认成功 → 重新发送请求" 无限循环
- ❌ Network标签显示大量 `ask` 请求
- ❌ 智能体显示红色错误："薪资数据访问需要二次确认"
- ❌ 用户无法获得答案

### 根因（Claude老师分析）

**核心问题**：确认成功拿到 `confirm_token` 后，重新发送的请求没有被后端识别为"已确认"，导致：

```
1. 用户填写事由，点击"确认并查看"
2. 前端获取 confirm_token ✅
3. 重新发送智能体请求
4. ❌ 后端收不到token，返回 reject(reason='需要二次确认')
5. ❌ 前端收到reject事件，再次触发确认流程
6. ❌ 因为TTL还有效，确认秒过，返回true
7. ❌ 又重新发送请求... 形成死循环
```

---

## 🔍 深层原因分析

### 原因1：缺少循环保护
每次收到 `reject` 事件（reason='需要二次确认'）都会触发 `handlePayrollConfirmForAgent`，没有检查是否已经在处理中。

**旧代码**:
```javascript
}else if(event==='reject'){
  if(data.reason === '需要二次确认'){
    msg.needsPayrollConfirm = true;
    msg.originalQuestion = q;
    // ❌ 没有检查是否已经在处理，直接触发
    setTimeout(() => handlePayrollConfirmForAgent(s, msg), 100);
  }
}
```

### 原因2：无重试限制
没有限制重试次数，可以无限循环。

### 原因3：Token传递问题（待验证）
虽然 `apiHeaders()` 应该添加 `X-Payroll-Confirm` 头，但可能存在：
- AUTH对象引用问题
- Token在不同作用域中不同步

---

## ✅ 修复方案

### 修复1：添加循环保护

**文件**: `frontend/index.html` - `handleAgentEvent` 函数

**新增逻辑**:
```javascript
}else if(event==='reject'){
  if(data.reason === '需要二次确认'){
    // ✅ 循环保护：如果已经在处理确认，不要重复触发
    if(msg.needsPayrollConfirm){
      console.error('[循环保护] 检测到重复的薪资确认请求，已拦截');
      msg.answer=`<div class="ans-reject"><b>薪资确认失败</b><br>Token传递异常，请刷新页面重试。</div>`;
      msg.done=true; msg.orchOpen=false;
      return;  // ← 关键：直接返回，不再触发
    }
    
    // 标记为正在处理
    msg.needsPayrollConfirm = true;
    // ... 触发确认
  }
}
```

### 修复2：添加重试计数器

```javascript
// ✅ 检查重试次数
msg.payrollConfirmRetry = (msg.payrollConfirmRetry || 0) + 1;
if(msg.payrollConfirmRetry > 1){
  console.error('[循环保护] 薪资确认重试次数超限');
  msg.answer=`<div class="ans-reject"><b>薪资确认失败</b><br>请刷新页面重试。</div>`;
  msg.done=true; msg.orchOpen=false;
  return;  // ← 最多重试1次
}
```

### 修复3：增强调试日志

**在 `handlePayrollConfirmForAgent` 中**:
```javascript
async function handlePayrollConfirmForAgent(s, msg){
  console.log('[智能体薪资] 开始处理薪资确认，重试:', msg.payrollConfirmRetry);
  console.log('[智能体薪资] 当前Token状态:', AUTH.payrollConfirmToken ? '有Token' : '无Token');
  console.log('[智能体薪资] TTL状态:', AUTH.payrollConfirmedUntil > Date.now() ? 'TTL有效' : 'TTL已过期');
  
  // ... 确认流程 ...
  
  if(ok && AUTH.payrollConfirmToken){
    console.log('[智能体薪资] 确认成功，准备重新发送请求');
    console.log('[智能体薪资] 检查apiHeaders:', apiHeaders());
    
    msg.needsPayrollConfirm = false;  // ✅ 标记为已处理
    // ... 重新发送请求
  }
}
```

### 修复4：确保 needsPayrollConfirm 标记正确清除

```javascript
// 确认成功后
msg.needsPayrollConfirm = false;  // ✅ 清除标记，允许下次确认

// 用户取消时
msg.needsPayrollConfirm = false;  // ✅ 也要清除
```

## 🔄 修复后的流程

### 正常流程
```
1. 用户问："张三的工资是多少？"
2. 后端返回 reject(reason='需要二次确认')
3. 前端触发 handlePayrollConfirmForAgent
   - msg.payrollConfirmRetry = 1
   - msg.needsPayrollConfirm = true
4. 用户填写事由，确认成功
   - 获取 token
   - msg.needsPayrollConfirm = false
5. 重新发送请求（带token）
6. 如果token有效 → 正常返回答案 ✅
   如果token无效 → 再次返回reject
7. 再次触发确认
   - msg.payrollConfirmRetry = 2
   - 超过限制（>1），拦截并报错 ✅
```

### 循环保护生效
```
1. 第一次触发确认：msg.needsPayrollConfirm = true
2. 如果再次收到reject：
   - 检测到 msg.needsPayrollConfirm = true
   - 直接拦截，显示"Token传递异常" ✅
   - 不再触发确认
```

## 🎯 修复要点

### 1. 双重保护机制
```javascript
// 保护1：检测是否正在处理
if(msg.needsPayrollConfirm){ return; }

// 保护2：检测重试次数
if(msg.payrollConfirmRetry > 1){ return; }
```

### 2. 标记的生命周期
```
初始状态：msg.needsPayrollConfirm = undefined
收到reject：msg.needsPayrollConfirm = true  (标记为处理中)
确认成功：msg.needsPayrollConfirm = false  (清除标记)
用户取消：msg.needsPayrollConfirm = false  (清除标记)
```

### 3. 重试计数器
```
首次reject：msg.payrollConfirmRetry = 1  (允许)
二次reject：msg.payrollConfirmRetry = 2  (拦截)
```

## ⚠️ 已知问题（待进一步验证）

虽然添加了循环保护，但**根本问题可能仍在Token传递**：

### 需要验证的点
1. **`apiHeaders()` 是否正确添加了 `X-Payroll-Confirm` 头？**
   - 检查方法：看控制台日志 `[API请求] 添加薪资确认头`
   - 检查方法：看Network标签，ask请求的Request Headers

2. **后端是否正确接收并验证了Token？**
   - 检查后端日志
   - 验证 `X-Payroll-Confirm` 头是否传到后端

3. **Token的作用域问题？**
   - `AUTH` 对象在 index.html 中定义
   - `window.AUTH` 暴露给 permission-admin.js
   - `apiHeaders()` 是否能正确读取？

### 如果循环保护触发了
说明Token传递确实有问题，需要：
1. 检查 `apiHeaders()` 函数
2. 检查 `window.AUTH` 与 `AUTH` 的同步
3. 可能需要使用 `window.AUTH.payrollConfirmToken` 而不是 `AUTH.payrollConfirmToken`

## 📝 总结

通过添加**循环保护**和**重试限制**，确保即使Token传递失败，也不会陷入无限循环。这是一个**防御性修复**，真正的根本解决还需要验证Token是否正确传递到后端。

**关键改进**：
- ✅ 循环检测：防止重复触发确认
- ✅ 重试限制：最多重试1次
- ✅ 详细日志：便于诊断问题
- ✅ 标记管理：正确的状态转换

**用户现在应该能够**：
- 填写确认表格后，要么正常获得答案
